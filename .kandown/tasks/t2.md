# Mutualiser FCM-Pi et créer `fcm-opencode`

## Résumé exécutif

Créer une architecture d’extensions FCM pour agents de codage où **FCM-Pi** et la future extension **`fcm-opencode`** partagent presque tout le code métier : scan des modèles, cache, accès daemon, ranking, résolution d’API keys, normalisation des endpoints, formatage des modèles et diagnostics.

L’objectif n’est pas de copier `pi-extension/` pour OpenCode. L’objectif est de transformer l’extension actuelle en un **core réutilisable + deux adapters ultra fins** :

```txt
FCM extension shared core
├─ scan daemon/direct
├─ rank models
├─ cache TTL
├─ API keys
├─ provider/model descriptor normalization
├─ config payload builders
└─ target adapters
   ├─ Pi adapter       → ExtensionAPI, ctx.ui, pi.registerProvider, pi.setModel
   └─ OpenCode adapter → Plugin hooks, config mutation, OpenCode provider schema
```

## Contexte actuel observé

### Extension existante

L’extension fraîchement créée est dans :

```txt
pi-extension/
├── package.json
├── README.md
├── extensions/index.js
└── lib/
    ├── api-keys.js
    ├── config-writer.js
    ├── daemon-client.js
    ├── direct-scanner.js
    ├── model-ranker.js
    └── scanner.js
```

Elle expose aujourd’hui :

- scan automatique au `session_start` Pi ;
- cache disque de 10 minutes dans `~/.pi/agent/fcm-cache.json` ;
- scan daemon via `http://localhost:19280/api/models` ;
- fallback direct avec `free-coding-models/src/core/ping.js` + `benchmark.js` ;
- ranking composite : SWE 60%, latency 20%, TPS 10%, stability 10% ;
- commandes Pi : `/fcm`, `/fcm-list`, `/fcm-router`, `/fcm-status` ;
- écriture de `~/.pi/agent/models.json`, `settings.json`, `auth.json` ;
- tentative d’auto-failover sur erreurs provider.

## Problèmes à corriger dans l’extension Pi actuelle avant mutualisation

- [ ] `ctx.ui.notify(..., 'warn')` devrait utiliser `warning` selon la doc Pi.
- [ ] `/fcm-router` appelle `pi.setModel({ provider, modelId })`, alors que Pi attend un vrai objet `Model` trouvé via `ctx.modelRegistry.find(...)`.
- [ ] `config-writer.js` duplique partiellement `src/core/endpoint-installer.js` et a une forme plus complète que `installIntoPi`; il faut centraliser la création de config Pi.
- [ ] `direct-scanner.js` mélange logique métier et rendu ANSI `chalk`; le core doit produire des événements de progression structurés, et l’adapter Pi décide du rendu.
- [ ] `package.json` de `fcm-pi` met `free-coding-models` en `peerDependencies`; pour une install npm autonome, il faudra probablement le mettre en `dependencies` ou déclarer clairement le setup requis.
- [ ] Les imports profonds `free-coding-models/src/core/...` doivent être validés dans un vrai `npm pack`/install, car ils sont sensibles au champ `files` du package root.
- [ ] Le cache est Pi-spécifique (`~/.pi/agent/fcm-cache.json`); il faut un cache shared avec compat de lecture de l’ancien chemin.

## Direction d’architecture retenue

### Structure cible recommandée

Garder `pi-extension/` en wrapper de compatibilité locale, mais déplacer le vrai code vers une structure de packages claire :

```txt
packages/
├── fcm-agent-core/
│   ├── package.json
│   └── src/
│       ├── index.js
│       ├── api-keys.js
│       ├── cache.js
│       ├── daemon-client.js
│       ├── direct-scanner.js
│       ├── ranker.js
│       ├── scan-orchestrator.js
│       ├── model-normalizer.js
│       ├── provider-config-builders.js
│       └── progress.js
├── fcm-pi/
│   ├── package.json
│   ├── README.md
│   └── extensions/index.js
└── fcm-opencode/
    ├── package.json
    ├── README.md
    └── src/index.ts

pi-extension/
├── package.json             # compat local path, imports packages/fcm-pi
├── README.md                # points to packages/fcm-pi
└── extensions/index.js      # tiny re-export wrapper
```

Alternative acceptable si on veut éviter un package npm core séparé au départ :

```txt
agent-extensions/
├── shared/
├── pi/
└── opencode/
```

Mais la cible long terme doit rester **un core publié/packable** pour éviter la duplication lors de la publication de `fcm-pi` et `fcm-opencode`.

Le core ne doit plus envoyer des strings déjà colorées. Il doit émettre :


- Pi adapter rend ça en status bar avec spinner + couleurs Pi/TUI.
- OpenCode adapter rend ça en logs, toast si possible, ou output de tool.

## Risques et décisions à valider

### Risque 1 — Hot-swap OpenCode

Pi expose `pi.setModel()`. OpenCode expose clairement la mutation de config au startup, mais le hot-swap runtime via plugin doit être vérifié. Ne pas promettre un switch instantané dans une session OpenCode tant que le SDK ne le confirme pas.

Décision proposée : pour v1, `fcm-opencode` met à jour `config.model` au startup et fournit `fcm_scan` pour préparer le prochain modèle. Si un switch runtime fiable existe, l’ajouter en v1.1.

### Risque 2 — Direct scan au startup

Le scan direct peut prendre 8–15s et déclencher beaucoup d’appels provider. C’est acceptable sur Pi si vava le veut, mais moins acceptable dans OpenCode `config` hook.

Décision proposée : OpenCode startup = cache/daemon only par défaut. Direct scan uniquement via `/fcm`/tool ou option explicite.

### Risque 3 — Publication multi-packages

Le repo actuel n’est pas structuré comme monorepo packages. Ajouter trois packages peut toucher le workflow npm.

Décision proposée : commencer avec dossiers packages et tests `npm pack`, sans publication séparée tant que vava n’a pas validé la forme finale.

## Notes d’implémentation pragmatiques

- Ne pas supprimer `pi-extension/` au premier refactor : garder compat local path pour les settings Pi de vava.
- Faire le core en JS ESM d’abord pour rester cohérent avec le repo actuel et éviter une toolchain TS supplémentaire. OpenCode adapter peut être TS seulement si on ajoute un build simple.
- Si TS OpenCode ajoute trop de friction, écrire `src/index.js` avec JSDoc types et garder `@opencode-ai/plugin` en dev/peer.
- Factoriser les builders de provider avant de toucher aux scanners : ça réduit le risque de casser l’app principale.
- Prévoir un mode `dryRun` pour OpenCode/Pi adapters afin de tester sans écrire dans les vrais fichiers utilisateur.

## Completion report — 2026-07-08 (v0.5.48)

### What landed

The shared-core extraction is complete. The repo now has a clean three-package layout under `packages/`:

```
packages/
├── fcm-agent-core/   ← scan, rank, cache, daemon, API keys, provider descriptors (pure, no rendering, no chalk)
├── fcm-pi/           ← Pi adapter (hooks, commands, status-bar renderer, ~/.pi/agent disk writer)
└── fcm-opencode/     ← OpenCode adapter (config mutation, commands, toasts, shell.env)
```

- **Core (`fcm-agent-core`)** owns all shared logic. It emits *structured* progress events (`{ phase, percent, completed, total, activeModels, message }`) — no ANSI/chalk. Provider descriptors are centralized: `buildPiProviderDescriptor`, `buildOpenCodeProviderDescriptor`, `buildSmartRouterDescriptor`, plus `normalizeBaseUrl` / `getProviderId`.
