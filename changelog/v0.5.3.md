# Changelog v0.5.3 - 2026-06-01

### Added
- **Web Dashboard M1 parity** — the local Web Dashboard now ships a feature set that mirrors the TUI for the everyday table flow. New: a `⌘K` command palette (the Web's only global keyboard shortcut), per-row ⭐ favorite button (persists through the same `~/.free-coding-models.json` file the TUI uses), clickable AI Lat. cell that fires a per-row AI Speed Test (TUI `Ctrl+A`), tri-state `auto / dark / light` theme cycle (TUI `G`), and a dedicated `Reset` button in the filter bar that counts active filters and resets them all in one click (TUI `N`).
- **New filter chips in the Web** — `Verdict` (9 states), `Health` (7 states), and a `Visibility` dropdown (Normal / Configured only / Usable only) that mirror the TUI's `V`, `H`, and `E` key cycles exactly. A `Custom text filter` chip appears whenever a sticky text filter is active, with an `X` to clear (TUI `X`).
- **Tool-compat dark-red row highlight** — when a tool mode is active, rows for models that aren't compatible with that tool now render with a dark red background, matching the TUI's render-table behavior. The tool mode picker itself ships in M3, but the visual is wired and ready.
- **Sticky FilterBar + sticky table header row** — the chip row (tier / status / verdict / health / visibility / provider / ping mode) now stays visible under the sticky header when scrolling a long table. The table's column-header row sticks to the top of the table's scroll area, just below the FilterBar.
- **URL deep-linking (read-only)** — `?view=dashboard&tier=S+&sort=verdict&origin=groq&…` hydrates the dashboard on load. This is the Web's answer to TUI CLI flags: shareable links instead of command-line arguments. M2 will add write-back so any filter change also updates the URL.
- **Favorites shared with the TUI** — the Web now reads and writes through the same `config.favorites` array the TUI uses (no separate localStorage). A new `/api/favorites` endpoint backs the `useFavorites` hook. The starred models, their priority order, and the pinned+sticky display mode are now consistent between surfaces.
- **Per-row reorder** — favorited rows can be moved up or down in router priority from the DetailPanel (TUI `Shift+↑↓`).
- **New `useFavorites` hook** — encapsulates the favorites read/write, the per-row isFavorite / rank / toggle / reorder / setPinnedAndSticky API, and the load error fallback.
- **New `useUrlState` hook** — parses the URL on mount and hydrates the active view from `?view=…`. Validation tables are exported so M2's write-back path uses the same allowlist.

### Changed
- **Removed the left sidebar.** All navigation now lives in the header. The model table uses 100% of the viewport width at every browser size — the user can resize the browser freely and the table reflows without ever being squeezed.
- **Removed the StatsBar** (5 cards above the table) — the data is reachable from the table, the filter chips, and the Analytics view; the cards were redundant.
- **Ping indicator** now always shows `next ping in Xs` (TUI style) instead of switching between `Pinging…` and a countdown. A small pulsing dot is the only "is actively pinging" hint.
- **ModelTable layout is now seamless** — no margin, no border-radius, no left/right border. The table flows edge-to-edge under the sticky header + sticky FilterBar.
- **Theme button** is now a tri-state cycle (auto / dark / light) with a clear label showing the current mode. Auto mode follows the OS-level `prefers-color-scheme` preference and updates live.
- **Header has 5 always-visible nav buttons** (Dashboard, Settings, Analytics, Recommend, Router) plus a kebab overflow menu (Help, Changelog, Install Endpoints, Installed Models). Items not yet shipped show an `M2` / `M3` / `M4` badge and trigger a friendly "coming soon" toast on click — the menu structure is honest and complete from day one.
- **Theme auto mode** listens to `prefers-color-scheme` changes and updates the resolved theme live.
- **README Web Dashboard section** expanded from 6 lines to a full feature table with M1 features, M2/M3/M4 roadmap badges, and explicit notes on what the TUI does that the Web does not (no keyboard shortcuts, full table coverage of every cycle).

### Why
The TUI has been the source of truth for the model catalog since day one, but most users land on the Web Dashboard first. M1 brings the everyday TUI flow to the Web — favorites, filters, sort, theme, per-row benchmark, command palette, table navigation — while the TUI keeps its keyboard-driven UX untouched. The Web gains a complete menu structure (with `M2/M3/M4` badges on the items not yet shipped) so the roadmap is visible at every step.
